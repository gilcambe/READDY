PARTE 1 — FILOSOFIA DO SITE BUILDER
O Site Builder da NEXIA OS é o coração da plataforma. É o que diferencia a NEXIA de um simples sistema: o dono do negócio ou o admin da NEXIA pode editar absolutamente TUDO no site do cliente — textos, imagens, cores, seções, ordem, vídeos, botões, formulários — sem precisar de programador, sem dar F5, sem depender de ninguém.


PRINCÍPIO FUNDAMENTAL:
NADA pode ser fixo no código. Tudo vem do Firestore. Tudo é editável via backend. Qualquer mudança reflete no site público em menos de 500ms via Firestore onSnapshot.


Pergunta
	Resposta
	Quem edita?
	O admin NEXIA Master (Gildivan) e o admin de cada tenant (Henrique da CES, etc.). Cada um com seus limites de acesso.
	Como salva?
	Auto-save com debounce de 600ms. Sem clicar em 'Salvar'. Apenas digitar e o sistema salva sozinho.
	Como aparece no site?
	Firestore onSnapshot() no frontend detecta mudança em <50ms e re-renderiza o DOM automaticamente.
	O visitante precisa de F5?
	Não. A mudança aparece em tempo real para quem já está no site.
	E o preview?
	Iframe do site real dentro do admin. O admin vê exatamente o que o visitante vai ver enquanto edita.
	Quem não pode editar?
	O HTML estrutural da CES (ces-landing.html) é protegido. Não pode mudar estrutura HTML, IDs, classes do template original.
	



1.1 Fluxo Técnico Completo: Admin Digita → Site Atualiza


#
	Etapa
	1
	Admin está na aba 'Editar Conteúdo'. Muda o título do hero para 'Nova Missão Empresarial'.
	2
	oninput='autoSave("hero")' é disparado a cada tecla.
	3
	autoSave() executa: clearTimeout(saveTimers['hero']); setTimeout(() => saveSection('hero'), 600). Timer reinicia a cada tecla — aguarda 600ms de silêncio.
	4
	600ms sem digitação: saveSection('hero') é chamado. Coleta todos os campos do form: {badge, title, subtitle, videoId, ctaText, sympla}.
	5
	setDoc(doc(db, 'tenants', 'ces', 'content', 'hero'), {...dados, updatedAt: serverTimestamp()}, {merge:true}). Escrita atômica no Firestore.
	6
	Firestore propaga a mudança para todos os subscribers em <50ms globalmente via WebSocket.
	7
	No site público (ces-app.js): db.doc('tenants/ces/content/hero').onSnapshot(snap => { state.hero = snap.data(); render(); }). O evento dispara.
	8
	render() executa: document.getElementById('app').innerHTML = renderNav() + renderHero() + renderAbout() + renderSpeakers() + renderInscricao() + renderFooter(). DOM atualizado.
	9
	Visitante que está na página ve o novo título sem dar F5. Em <500ms do momento em que o admin parou de digitar.
	10
	Toast no admin: 'Salvo automaticamente ✓'. Admin tem confirmação visual de que o Firestore recebeu a escrita.

PARTE 2 — SITE BUILDER DA NEXIA MASTER
O admin NEXIA (Gildivan) tem um Site Builder diferente dos clientes. Ele gerencia a landing page da própria NEXIA (/index.html) e pode criar/editar a landing de qualquer tenant.


2.1 O que o Master pode editar — Site NEXIA


Seção
	O que é editável
	SEÇÃO HERO
	Título principal, subtítulo, badge de destaque, texto do botão CTA principal, link do CTA, botão secundário, vídeo/imagem de fundo.
	SEÇÃO CLIENTES (Tenant Grid)
	Cards dos 4 clientes atuais: nome, ícone RemixIcon, cor, descrição, tags de módulos. Ordem dos cards (drag-and-drop). Adicionar novos tenants ao grid.
	SEÇÃO SOBRE / FEATURES
	4 cards de diferenciais: ícone, título, texto. Adicionar/remover/reordenar cards. Trocar ícones da biblioteca RemixIcon.
	NAVBAR
	Logo NEXIA, links de navegação (textos + anchors), botão 'Acessar Sistema' (texto + link).
	FOOTER
	Texto de copyright, links do footer (Login, Master, Dev Viewer), email de contato.
	BRANDING GLOBAL
	Cor primária (#00E5FF), cor secundária (#7B5CFA), fonte padrão, logo URL, favicon URL. Aplicado em todo o ecossistema.
	META/SEO
	Title da página, meta description, OG title, OG description, OG image URL, robots.
	SCRIPTS/PIXELS
	Google Analytics ID, Facebook Pixel ID, Google Tag Manager ID, Hotjar ID. Campos limpos para colar os IDs.
	STATUS PAGE
	Texto de status de cada serviço, horário de atualização, mensagem de incidente ativo.
	

2.2 O que o Master pode editar — Qualquer Tenant
Via Tenant Control ou Shadow Login, o Master pode editar o conteúdo de qualquer tenant como se fosse o admin daquele cliente:


* Todos os campos do editor do tenant (hero, sobre, formulário, rodapé, branding)
* Criar seções novas no site do cliente
* Ativar/desativar seções existentes
* Resetar conteúdo para o template padrão do setor
* Publicar mudanças emergenciais sem precisar do admin do tenant

	ctaLink
	Link do botão CTA. Padrão: '#inscricao' (ancora na seção). Pode ser URL externa (Sympla, etc.).
	sympla
	URL completa do evento no Sympla. Se preenchido, aparece botão secundário 'Ver no Sympla →'. Se vazio, botão some.
	Cor Primária (via Settings)

3.3 Aba SOBRE — Todos os Campos


Campo
	Descrição e Regras
	title
	Título da seção 'Sobre'. Aparece centralizado em h2. Exemplo: 'O que é a Missão CES?'.
	text
	Parágrafo explicativo. Texto corrido. Máx 400 chars. Aparece centralizado em 17px com max-width 600px.
	items[0..N] — icon
	Ícone do card. Classe RemixIcon. Exemplo: 'ri-global-line'. Ver biblioteca em remixicon.com.
	items[0..N] — title
	Título do card de destaque. Exemplo: 'Las Vegas'. Aparece em h3.
	items[0..N] — desc
	Descrição do card. Exemplo: 'Janeiro 2027 — CES International'. Aparece em 14px.
	Botão Adicionar Card
	Cria novo item com ícone padrão ri-star-line. Admin preenche os 3 campos.
	Botão Remover Card
	Remove item pelo índice. O array é reenumerado automaticamente.
	Quantidade de cards
	Mínimo 1, máximo sem limite (layout responsivo em grid auto-fill 280px).

3.7 Seção CONFIGURAÇÕES DE BRANDING CES


Campo

3.8 Seção UPLOAD DE MÍDIA


Campo/Função
	Descrição
	Tipos aceitos
	Imagens: image/* (JPG, PNG, GIF, WebP, SVG). Vídeos: video/* (MP4, WebM). Documentos: .pdf. Planilhas: .xlsx, .csv.
	Interface
	Área com bordas tracejadas. Clique ou arraste. Input file type=file multiple oculto acionado por click.
	Processo de upload
	1. Mostra 'Enviando {nome}…' no progress. 2. storageRef(storage, 'tenants/ces/uploads/{timestamp}_{filename}'). 3. uploadBytes(). 4. getDownloadURL(). 5. Card na galeria.
	Galeria de arquivos
	Grid responsive. Imagens: thumbnail 140px + nome + botão 'Copiar URL'. Outros: ícone de arquivo + nome + botão 'Copiar URL'.
	Copiar URL
	navigator.clipboard.writeText(url). Toast 'URL copiada!'. Admin usa essa URL nos campos de imagem do editor.
	Caminho no Storage
	tenants/ces/uploads/{timestamp}_{filename}. Organizado por tenant. Visualizável no Firebase Console.
	Sem delete no MVP
	A galeria não tem botão de deletar por padrão (para evitar quebrar links). A exclusão é feita via Firebase Console.
	Limite de tamanho
	Firebase Storage free tier: 1GB total. Sem limite por arquivo. Recomendado: imagens <2MB, vídeos <50MB.
	

3.9 Seção PRÉVIA DO SITE


Elemento
	Descrição
	Iframe
	<iframe src='/tenants/ces/index.html' width='100%' height='500px'>. Renderiza o site real.
	Atualização
	O iframe tem seus próprios listeners onSnapshot ativos. Quando conteúdo muda no Firestore, o iframe atualiza sozinho sem recarregar.
	Responsividade
	O admin pode ver como fica em desktop (largura total) e mobile (reduzir o browser ou adicionar botões de device). A prévia mostra o estado exato do site público.
	Limitação
	Iframes com mesmo domínio funcionam perfeitamente. Cross-domain pode bloquear por segurança do browser (CORS/X-Frame-Options).

PARTE 8 — EDITOR VISUAL AVANÇADO (WYSIWYG)
Além do editor por formulário, o Site Builder da NEXIA deve ter um modo de edição visual direta — WYSIWYG (What You See Is What You Get) — onde o admin clica diretamente no texto do site e edita inline.


8.1 Como Funciona o Modo WYSIWYG


Aspecto
	Como funciona
	Ativação
	Botão 'Modo de Edição' no admin. Abre o site em um iframe com borda de edição.
	Elementos editáveis
	Textos (h1, h2, h3, p, span, a): contentEditable=true ao ativar. Clique no texto → cursor de edição.
	Imagens
	Clique na imagem → aparece botão 'Trocar imagem'. Abre galeria do Storage.
	Cores
	Click em elemento colorido → seletor de cor inline.
	Auto-save
	Ao fazer blur do elemento editado: captura o novo valor e chama saveSection(). Debounce 1s.
	Indicadores visuais
	Elementos editáveis têm borda tracejada azul ao hover. Tooltip 'Clique para editar'.
	Blocos
	Cada seção tem handle de drag para reordenar. Drop → atualiza campo 'order' no Firestore.
	Resultado
	O que o admin vê NO iframe É exatamente o que o visitante vai ver. Zero discrepância.
	

8.2 Gestor de Blocos — Adicionar/Remover/Reordenar Seções


Recurso
	Como funciona
	Painéis de seção
	Cada seção do site aparece como um bloco no painel lateral: Hero, Sobre, Palestrantes, Inscrição, Footer. Com olho (visível/oculto) e handle de reordenação.
	Visibilidade
	Toggle eye icon: seção some do site público sem apagar os dados. Firestore: section.visible = false.
	Reordenação
	Drag handles. Nova ordem salva em Firestore como array de {id, order}. Render usa order para montar o HTML.
	Adicionar nova seção
	Botão '+ Adicionar Seção'. Modal com tipos: Texto simples, CTA Band, Galeria de fotos, Vídeo embed, Depoimentos, Stats, Mapa, Iframe.
	Deletar seção
	Botão lixeira com confirm(). Remove do Firestore. Site atualiza via onSnapshot.
	Seções obrigatórias
	Hero e Footer não podem ser deletados. Botão lixeira desabilitado.
	

8.3 Tipos de Seção que Podem Ser Adicionadas


Tipo de Seção
	Campos CMS disponíveis
	Texto Simples
	Título (h2), texto (p), opção de fundo claro/escuro. CMS fields: title, text, background.
	CTA Band
	Faixa com chamada para ação. CMS: title, subtitle, buttonText, buttonLink, background color.
	Galeria de Fotos
	Grid de imagens do Storage. CRUD de imagens com caption opcional. Layout: grid 3/4 cols.
	Vídeo Embed
	Embed YouTube/Vimeo por ID. CMS: videoId, platform (youtube/vimeo), title, autoplay (bool), muted (bool).
	Depoimentos
	Cards de clientes. CRUD: texto, nome, cargo, empresa, foto, nota (1-5 estrelas).
	Stats/Números
	Row de 4 métricas com números grandes. CMS: [{number, label, suffix}]. Ex: '3.400 Clientes', '98% NPS'.
	Mapa
	Embed Google Maps por coordenadas ou endereço. CMS: address, zoom, latitude, longitude.
	Iframe Externo
	Embed de qualquer URL via iframe. CMS: url, height (px), title.
	FAQ Accordion
	Perguntas e respostas expansíveis. CRUD de {question, answer}. Auto-save ao editar.
	Timeline
	Lista de marcos com ícone, data e texto. Para eventos, jornadas, histórico da empresa.
	Pricing Cards
	Cards de planos com preço, features, CTA. CRUD completo.
	Formulário Customizado
	Construtor de form: adicionar campos (text, email, tel, select, textarea, checkbox). Destino: Firestore ou webhook.

PARTE 9 — CONTROLES DE PUBLICAÇÃO E VERSIONAMENTO


9.1 Sistema de Draft vs. Publicado


Estado
	O que significa
	Draft (Rascunho)
	Conteúdo editado mas não publicado. Salvo em tenants/{id}/content/{section}/draft. Visível apenas no admin.
	Publicado
	Conteúdo aprovado. Salvo em tenants/{id}/content/{section}. Visível no site público.
	Botão 'Publicar'
	Copia conteúdo do draft para o publicado. addDoc(_audit_logs, action:'content.published').
	Botão 'Descartar Rascunho'
	Deleta draft. Restaura visualização do publicado atual.
	Preview de Draft
	No admin: iframe carrega com parâmetro ?mode=draft. Site lê do path draft em vez do publicado.
	Benefício
	Admin pode trabalhar em mudanças grandes (redesign de hero, nova seção) sem afetar o site ao vivo até estar pronto.
	

9.2 Histórico de Versões


Recurso
	Como funciona
	Snapshot automático
	A cada publicação: cópia do estado atual salva em tenants/{id}/versions/{timestamp}. Até 50 versões guardadas.
	Visualizar versão
	Admin seleciona versão no histórico. Preview no iframe mostra como o site estava naquele momento.
	Restaurar versão
	Botão 'Restaurar esta versão'. Copia dados da versão para content/{section}. Publicação imediata.
	Campos da versão
	{timestamp, publishedBy, sections: {...todo o conteúdo no momento}}. Snaphsot completo.
	Limite
	Firestore guarda até 50 versões por tenant. A 51ª deleta a mais antiga.
	

9.3 Status da Publicação no Admin


Indicador
	Quando aparece
	Indicador 'Salvo'
	Toast verde no canto inferior. Aparece por 2.5s após cada autoSave().
	Indicador 'Publicando…'
	Spinner durante o setDoc(). Botão desabilitado.
	Indicador 'Publicado'
	Timestamp 'Última publicação: X min atrás' no topo do editor.
	Indicador 'Rascunho não publicado'
	Badge amarelo no título da seção quando há draft pendente.
	Indicador de mudança não salva
	Se perder conexão antes do debounce: indicador vermelho 'Alterações não salvas'. Tenta reenviar ao reconectar.

10.1 Previews no Admin


Device

10.2 CSS Responsivo do Site Público


Regra CSS
	O que faz
	@media (max-width: 768px)
	Padding de seções reduz de 100px para 60px. Navbar padding reduz para 16-20px.
	@media (max-width: 480px)
	Título do hero reduz para clamp(32px, ...). Grid de cards vira 1 coluna.
	Navbar mobile
	Links de navegação somem. Ícone de menu hamburger aparece (a implementar menu mobile).
	Formulários
	Full width em mobile. Grids 2 colunas viram 1 coluna.
	Fontes responsivas
	Todos os títulos usam clamp(). Exemplo: clamp(40px, 6vw, 76px) — nunca abaixo de 40px, nunca acima de 76px.
	Imagens
	Sempre width:100%, height fixo com object-fit:cover. Nunca overflow horizontal.
